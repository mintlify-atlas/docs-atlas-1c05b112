## Core payment flows

Hyperswitch models every payment as a state machine. The four primary operations are authorize, capture, void, and refund.

### Authorize

An authorization reserves funds on the customer's card without settling them. You create a payment with `confirm: true` to authorize immediately, or `confirm: false` to create the payment intent and confirm it in a separate step.

```bash
curl --request POST \
  --url https://sandbox.hyperswitch.io/payments \
  --header 'Content-Type: application/json' \
  --header 'api-key: <your-api-key>' \
  --data '{
    "amount": 6540,
    "currency": "USD",
    "confirm": true,
    "capture_method": "manual",
    "customer_id": "cus_xyz",
    "payment_method": "card",
    "payment_method_data": {
      "card": {
        "card_number": "4242424242424242",
        "card_exp_month": "10",
        "card_exp_year": "25",
        "card_holder_name": "Joseph Doe",
        "card_cvc": "123"
      }
    },
    "return_url": "https://example.com/thank-you"
  }'
```

### Capture

When `capture_method` is `manual`, you call the capture endpoint once the goods or service are ready to be fulfilled. The captured amount can be equal to or less than the authorized amount.

```bash
curl --request POST \
  --url https://sandbox.hyperswitch.io/payments/<payment_id>/capture \
  --header 'Content-Type: application/json' \
  --header 'api-key: <your-api-key>' \
  --data '{ "amount_to_capture": 6540 }'
```

### Void

Cancel an authorized payment before capture. Once voided, the hold on the customer's funds is released.

```bash
curl --request POST \
  --url https://sandbox.hyperswitch.io/payments/<payment_id>/cancel \
  --header 'api-key: <your-api-key>' \
  --data '{ "cancellation_reason": "customer_initiated" }'
```

### Refund

Issue a full or partial refund against a captured payment. Hyperswitch routes the refund to the same processor that processed the original payment.

```bash
curl --request POST \
  --url https://sandbox.hyperswitch.io/refunds \
  --header 'Content-Type: application/json' \
  --header 'api-key: <your-api-key>' \
  --data '{
    "payment_id": "<payment_id>",
    "amount": 2000,
    "reason": "customer_initiated"
  }'
```

## How the orchestration layer works

The Hyperswitch routing engine evaluates every incoming payment against your configured routing rules and produces an ordered list of eligible processors. It then attempts the payment sequentially until one succeeds or all candidates are exhausted.

```
Incoming payment request
        │
        ▼
┌───────────────────┐
│  Active routing   │  Rule-based / Volume-based / Auth-rate-based / Least-cost
│  algorithm        │
└────────┬──────────┘
         │  Ordered processor list
         ▼
┌───────────────────┐
│ Eligibility check │  Payment method enabled? Card network supported?
└────────┬──────────┘
         │  Eligible connectors
         ▼
┌───────────────────┐
│   Attempt #1      │  Send to primary connector
└────────┬──────────┘
    fail │
         ▼
┌───────────────────┐
│  Smart Retry /    │  Step-up 3DS, alternate connector, clear-PAN, etc.
│  Fallback         │
└────────┬──────────┘
    pass │
         ▼
    Payment success / terminal failure
```

<Note>
  Hyperswitch appends an attempt suffix to your `payment_id` when retrying (e.g. `pay_abc_1`, `pay_abc_2`). Reconciliation remains tied to the original `payment_id` you supplied.
</Note>

## Payment status lifecycle

| Status | Meaning |
|--------|---------|
| `requires_payment_method` | Payment created, no method attached yet |
| `requires_confirmation` | Payment method attached, awaiting confirmation |
| `requires_customer_action` | Waiting for 3DS or redirect completion |
| `processing` | Sent to processor, awaiting response |
| `succeeded` | Payment authorized and (if auto-capture) settled |
| `partially_captured` | Partial capture applied |
| `cancelled` | Voided before capture |
| `failed` | All retry attempts exhausted |
| `requires_capture` | Authorized; awaiting manual capture |
